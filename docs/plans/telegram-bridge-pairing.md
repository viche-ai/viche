# Telegram Bridge Pairing Plan

## Scope

Implement Telegram bot integration for Viche using UUID-backed agents, durable Telegram identity mappings, and a browser-based pairing flow that claims the agent and joins one or more registries.

## Decisions

- Keep Viche agent IDs as UUIDs generated by the existing agent registration flow.
- Persist Telegram identity separately from agent records.
- Start Telegram-registered agents as unclaimed/public and keep them online until explicit deregistration.
- `/pair` creates a DB-backed one-time token.
- Pairing in the browser both claims the agent, removes it from `global`, and joins selected registries.
- Pairing is disabled when account auth is disabled on the instance.
- Only one active pairing link exists per Telegram agent at a time.
- Do not add app-wide admin roles or Telegram-specific admin endpoints.
- Do not accept API tokens or registry tokens pasted into Telegram for private-registry access.

## Architecture

1. Add Telegram persistence:
   - `telegram_agent_links`
   - `telegram_pairing_tokens`
2. Add Telegram context and schemas.
3. Add claim operation for unowned agents.
4. Add Telegram API wrapper and bridge GenServer.
5. Add browser pairing flow with authenticated handoff.
6. Add tests for context, bridge, and pairing flow.

## Flow

1. Telegram user runs `/start`.
2. Bridge collects display name and capabilities.
3. Bridge registers a normal UUID-backed Viche agent and stores Telegram link metadata.
4. Telegram user runs `/pair`.
5. Bridge creates one-time pairing token and returns browser link.
6. Browser flow authenticates the user if needed.
7. User confirms claim and selects multiple registries.
8. Server claims the agent, removes `global`, and joins the selected registries.

## Risks

- Claim must be atomic and only succeed while `user_id` is `nil`.
- Pairing tokens must be one-time and expire cleanly.
- Reply routing can stay in-memory, but durable Telegram links must survive restarts.
- Keep the feature aligned with the existing user-driven auth model and avoid introducing a parallel admin surface.

## Verification

- Migration and schema tests
- Telegram API wrapper tests with `Req.Test`
- Bridge tests for registration, forwarding, reply routing, `/pair`, and `/stop`
- Pairing controller/live tests for login handoff, claim, and multi-registry join
- `mix precommit`
